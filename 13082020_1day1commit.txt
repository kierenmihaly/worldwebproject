데이터 모델의 개념 -**현실정보를 컴퓨터에 표현 → 단순화, 추상화 → 개념적 모형**

- 데이터 모델의 구성요소
    - **개체(표현하려는것), 속성(필드), 관계(개체와 개체사이 연결)**
- 개념적 데이터 모델 - 인간적 이해
- 논리적 데이터 모델 - 컴퓨터이해
- 논리적 데이터모델의 품질 검증
    - 개체(학생,교수), 속성(학생의 이름,전화번호), 관계(개체와 개체사이에 어떤게 연결되어있는지), 식별자(명칭, 정의,부서), 전반적인 품질 검증항목
- 데이터 모델에 표시할 요소
    - 구조(성질), 연산(실제로 데이터를 처리하는 작업), 제약조건(실제 데이터의 논리적 제약조건)

데이터 모델의 구성요소 - 개체(Entity)

- 개체의 정의 및 특징 (디비에서 표현하려는것 개념, 정보단위, 현실세계 대상) - 교수개체- 유일란 식별자에의해 식별가능- 다른개체와 관계가 있다 , 학생개체

    교수번호, 성명, 전공, 소속등은 하나의 속성이다 

- 개체 선정 방법 - 어떻게 개체를 정의할거냐? , 회사마다 업무기술서가 있다. 그엄무를 담당하는 담장자와 인터뷰를 해야한다. 장부, 전표, DFD ⇒ 자료저장소- 자료관리를 그래프로 만들어놓음 ,, 판매자가 상품등록을한다 무슨일을 하는지 , 상품대장에 품목이있는데 등이
- 개체명 지정 방법 - 업무에서 사용 용어, 약어사용 x, 모든개체는 사용하는 단어가 다름

데이터 모델의 구성 요소 - 속성(attribute)

- 속성의 정의 및 특징 ;  작은단위, 항목, 특성, 속성수→ 차수,디그리
- 속성의 종류
    - 속성의 특성에 따른 분류 ; 기본(업무분석), 설계(모델링위해 업무를 규칙화 하려고), 파생(다른속성→ 영향받아)
    - 개체 구성방식에 따른 분류 ; 기본키, 외래키(다른개체), 일반(기본키,외래키 아닌거)
- 속성 후보 선정 원칙 - 많이 선정하는게 좋음
- 속성명 지정 원칙 - 정확하고 혼란이 없게 !
    1. 해당업무에서 사용하는 단어 2. 서술 x 3. 약어 x 4.객체명은 속성명으로 사용 x

데이터 모델의 구성요소 - 관계

- 관계의 정의 - 논리적 연결 , 교수 - 지도 - 학생
- 관계의 형태
    - 일대일, 일대다, 다대다
- 관계의 종류
    - 종속 관계(주 종), 중복 관계(2개의 개체사이에 2번이상 종속관계), 재귀 관계(개체가 자신과 관계를 갖는것), 배타 관계(속성 구분자 → 개체의 특성을 분할하는 관계, 배타 AND, 배타 OR)

식별자 - 유일하게 구분하는 거!  여러명의 학생중에 한명만 식별할 수 있는거 ! , 당연히 한개이상의 식별자를 가짐 

- 대표성 여부 - **주 식별자, 보조 식별자**
- 스스로 생성 여부
- 단일 속성 여부
- 대체여부

- 주 식별자 - 개체를 대표하는 유일한 식별자 예) 주민번호, 카드번호

    반드시 하나만 있어야함!!

    - 보조 식별자 - 주 식별자를 대신하는것 , 예를들어 주민번호, 카드번호, 사원번호 각각 주 식별자이지만 주 식별자는 하나만 있어야하니까 주민번호를 주 식별자라고 하면 나머지 카드번호와 사원번호는 보조 식별자임

**주 식별자는 유일성(유일하게 하나만), 최소성(주민번호 카드번호 둘다 불러도 되지만 둘중 하나만 부르는게 효율적!), 불변성, 존재성 (반드시 데이터 값이 있어야함)!** 

- **내부 식별자(개체 내)//외부식별자 (다른개체 와 연결되는거)**
- **단일식별자(1개속성)//복합식별자 (2개이상 속성)**
- 원조식별자(가공되지않은 식별자)//대리식별자(식별자 속성이 2개 있는게 있다. 이름하고 신청날짜 중복될 수 있는데 이걸 통쨰로 하나로 써야함 )
- **후보식별자 - 각 인스턴스를 유일하게 식별할 수 있는거 1개이상의 후보 식별자가 존재한다, null값은 비워둠, 자주 변경되면 안됨 , 예로 고객번호 주민번호 사원번호 있으면 3개 모두 후보식별자가 됨. 이중에서 주식별자가되고 나머지는 보조식별자가 됨**

    예) table1 고객- 고객번호, 이름, 전화번호, 주소 

    table2 주문 - 고객번호, 주문번호

- **관계형 데이터베이스의 Relation구조**
    - 튜플 ; 나디날리티, 기수
    - 속성 ; 예) 학번, 이름, 학년, 성적

    - 도메인 ; 속성이 취할 수 있는 . 예를들어 초등학년 123456학년 있는데 학년의 도메인은 123456입니다. 라고함 7은 없음

    튜플은 모두 다르다(같은게 있을 수 도있다) , 순서의 의미가 없다, 속성값은 원자값으로 들어와야한다.
